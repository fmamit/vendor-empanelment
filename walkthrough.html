☑ DPDP Consent

Before any data is collected, the vendor must provide explicit consent as per the Digital Personal Data Protection Act.

  • Consent checkbox with full privacy policy text
  • A consent_records row is created storing: vendor identifier, consent version, purpose, IP address, user agent
  • Consent can be withdrawn later — vendor status changes to consent_withdrawn

📞 Contact Details & OTP Verification

Both phone and email must be verified via OTP before proceeding.

Phone OTP (WhatsApp)

  • Enter 10-digit Indian mobile number
  • OTP sent via Exotel WhatsApp API
  • 6-digit code, expires in 5 minutes
  • Max 3 attempts per session
  • 60-second cooldown between resends
  • Rate limit: 5 OTPs per phone per hour

Email OTP

  • Enter valid email address
  • OTP sent via Resend with branded template
  • Same 6-digit code, 5-minute expiry
  • Max 3 attempts per session
ℹ️
Test Mode: When WhatsApp is not configured, the system returns a test OTP in the response. Phone +919999999999 accepts code 123456 as a bypass.

📄 Document Upload

Required documents depend on the selected vendor category. Each category has a configured list via category_documents.

  • Accepted formats: PDF, JPG, PNG
  • Max file size: 10 MB per document
  • Files stored in Supabase Storage bucket: vendor-documents
  • Path: referral/{identifier}/{docTypeId}/{timestamp}_{filename}
  • Each upload immediately triggers AI document analysis (Google Gemini 2.5 Pro)
  • Mandatory documents marked with * must be uploaded before submission

Verification APIs

Six real-time verification checks via Surepass. All results stored in vendor_verifications.

ℹ️
Provider: All verification APIs are powered by Surepass. Each request includes an Authorization: Bearer header. Retry logic with exponential backoff (up to 2 retries) is built in.

P PAN Verification Working

Endpoint: POST /api/v1/pan/pan
Edge Function: verify-pan
Input: pan_number, vendor_id
Validation: 10-char alphanumeric PAN format
Returns:
  • Registered name
  • Date of birth
  • Validation status (is_valid)
// Sample response { "success": true, "data": { "pan": "ABCDE1234F", "name": "RAJESH KUMAR", "dob": "1990-01-15", "is_valid": true } }

G GST Verification Working

Endpoint: POST /api/v1/corporate/gstin
Edge Function: verify-gst
Input: gstin, vendor_id
Validation: 15-char GSTIN regex
Returns:
  • Business name / legal name
  • Trade name
  • Registration date & status
  • Validation status
// Sample response { "success": true, "data": { "gstin": "27AABCU9603R1ZM", "business_name": "ACME CORP PVT LTD", "trade_name": "ACME CORP", "status": "Active", "registration_date": "2017-07-01", "is_valid": true } }

Returns:

A Aadhaar Verification (DigiLocker) Multi-step

Aadhaar verification is a two-step process using DigiLocker integration.

Step 1 — Initiate

Function: verify-aadhaar

Endpoint: POST /api/v1/digilocker/initialize

Input: vendor_id

Returns: client_id, token, and verification_id

Initializes the Surepass Digilocker SDK for the user.

Step 2 — Store Details

Function: get-aadhaar-details

Endpoint: Stores Aadhaar data from SDK callback

Input: verification_id, aadhaar_data

Returns: Success confirmation

Called after user completes DigiLocker auth via the SDK.

Security & Compliance

PII encryption, DPDP compliance, RLS, and audit logging.

PII Encryption

  • All sensitive fields encrypted with pgp_sym_encrypt
  • Encryption key stored in Supabase Vault
  • Auto-encrypt triggers on INSERT/UPDATE
  • Decrypted views (vendors_decrypted) for authorized access only
  • get_vendor_decrypted() RPC with auth checks & audit logging
🔒
Encrypted fields: PAN, GST, phone, email, bank account, IFSC, nominee contact, CIN number

Row-Level Security (RLS)

  • RLS enabled on all tables
  • Role-based policies: admin, approver, checker, maker, vendor
  • Helper functions: has_role(), is_internal_staff(), is_vendor_user()
  • Storage bucket RLS: folder-based vendor isolation
  • Public read access only for: categories, document types

DPDP Compliance

  • Consent tracking: version, purpose, timestamp, IP, user-agent
  • Data requests: vendors can request data export
  • Consent withdrawal: changes vendor status, stops processing
  • Breach notifications: tracked & sent to affected vendors
  • Admin audit page: /admin/dpdp-audit

Audit Logging

  • pii_access_log — who accessed what PII, when, and why
  • workflow_history — every status change with actor & comments
  • whatsapp_messages — all OTP messages sent
  • consent_records — consent given/withdrawn timeline
  • All verification request/response data stored as JSONB
